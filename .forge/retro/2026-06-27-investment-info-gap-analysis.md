- Divergences:
  - 계획서에 없던 **메타 발견**: "❌미보유"로 잡힌 갭 다수가 실은 *데이터는 이미 손안에 있는데 안 꺼내 쓰는* 것이었다 — KR 재무 건전성 지표는 Naver finance API의 미추출 row(순이익 row2·영업이익률 row5·순이익률 row6·ROE row7·부채비율 row8·당좌비율 row9), US는 yfinance의 미사용 cash_flow/balance_sheet/income_stmt 필드, 가격·기술은 `get_support_resistance`(함수만 있고 리포트 미연결). → 최우선 빌드 후보(재무 건전성 보강)가 신규 소스 연동이 아니라 "기존 데이터 배선"이라 가성비 최고로 판명. 이게 우선순위 종합의 핵심 축이 됨.
  - 워크플로우 분담: 8축 감사만 서브에이전트(sonnet), **축 간 우선순위 종합·문서 합성은 메인(opus)**에 둠(전역 일관성·판단 필요 작업이라). 계획서는 방법 미규정 — 자연스러운 분담.

## Learnings
- Do differently next time:
  - **서브에이전트-그라운딩 감사 결과는 "확정 사실"이 아니라 "빌드 출발점"으로 다룰 것.** 특히 외부 API의 *행/필드 인덱스 주장*(KR Naver row5=영업이익률 등)은 서브에이전트가 "직접 확인"이라 해도 빌드 착수 그릴링에서 실응답 재검증을 DoD에 넣어야 한다 — row 오인은 ×100 오저장류 사고(`wrong<missing`). 보고서/run.md에 caveat로 명시해 둠.
  - **갭 분석류 작업엔 "병렬 축별 감사(sonnet) → 메인 합성(opus)" 패턴이 잘 맞는다.** 8축 ~5분·656k 토큰으로 코드 그라운딩 완료. 각 서브에이전트에 CONTEXT 정본 구분을 명시 주입한 게 오라벨(매크로 신호↔상관 등) 방지에 유효했음 — 도메인 distinction이 풍부한 코드베이스 감사 시 재사용할 프롬프트 기법.
  - 데이터 가용성에 "확인 필요"가 붙은 갭(KR FnGuide EV/EBITDA 항목·KR Naver CAPEX row·KR 시장지수 밸류 소스)은 해당 후속 작업 그릴링에서 소스 실존을 먼저 검증할 것.
