## Learnings
- Do differently next time:
  - **추출 정밀화 키워드는 추측 말고 실제 적재 데이터 빈도로 고른다**: `regexp_matches(raw_text, '수주[가-힣]{1,3}', 'g')` 빈도 조회로 수주잔고(158)·수주총액(114)·수주잔액(18)=지표, 수주상황(327)·수주산업(183)·수주계약(156)=노이즈를 식별해 정탐 세트를 데이터로 확정. 텍스트 추출/필터 정밀화 작업의 일반 패턴 — 운영 데이터가 있으면 그걸로 키워드를 정하라.
  - **운영 데이터로 코드 변경을 검증한 효과**: 단위 일관성(19/20 단일)·노이즈 제거를 실적재 재실행으로 확인 — fixture 테스트만으론 못 잡는 실세계 품질을 드러냄. 데이터 정제 작업은 배포 후 재적재 UAT를 DoD에 넣을 것.
- 잔여/후속(저위험):
  - **"수주잔고" 용어 중의성**: 정기보고서에서 "수주잔고"는 (a) 수주잔고 지표 문장 + (b) **무형자산 회계 계정명**(영업권/수주잔고/기타무형자산 표, PPA 인식분)으로 둘 다 등장. 키워드만으론 분리 불가 → 현대건설만 백만원+억원 잔존. 실제 지표 문장이 raw_text 첫 줄이라 Cowork가 선택 가능. 더 좁히려면 무형자산/영업권 문맥 표 제외(후속 후보).
  - 다음 단계: Cowork가 `GET /api/report/backlog/pending`(현재 131행/20종)을 읽어 실제 수치를 `PUT`으로 채우면 숫자가 표시됨(원래 "채워줘"의 최종 완수).
