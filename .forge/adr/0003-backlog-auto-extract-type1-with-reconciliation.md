# 수주잔고 수치는 유형1(수주상황 표)을 코드가 구조파싱+검산으로 자동 채운다 (Cowork는 폴백)

ADR-0002는 회사별 형식 다양성을 이유로 자동 파싱(Option A)을 기각하고 수치를 Claude Cowork가 채우게 했다. 그러나 20개 KR 종목의 실제 `raw_text`를 검증한 결과, (a) "수주상황" 표는 헤더 컬럼(수주총액·기납품액·수주잔고, 또는 기초·신규·기납품·기말)이 사실상 표준화돼 있고 (b) `수주잔고 ≈ 수주총액 − 기납품액`(또는 `기말 = 기초 + 신규 − 기납품`) 산식으로 **자가 검증**이 가능하다는 것이 드러났다. 그래서 **유형1(수주상황 표)에 한해** 셀을 보존해 헤더로 `수주잔고 금액` 컬럼을 매핑·추출하고, **상대 1% 검산을 통과한 값만 `source='dart'`로 자동 저장**한다. 기각됐던 Option A와 다른 점은 (1) 납작화 정규식이 아니라 **셀 보존 + 헤더 컬럼 매핑**, (2) **검산 게이트** — 오추출은 산식이 안 맞아 자동으로 pending으로 떨어지므로 "잘못된 값 < 누락" 원칙이 코드로 강제된다는 것이다.

## Considered Options

- **회사별 정밀 파서** (Cowork의 최초 제안) — 기각. 표는 사실상 2~3개 패턴으로 표준화돼 있어 회사별 파서는 과잉이고 ADR-0002 Option A의 brittleness를 재현한다.
- **피드만 개선하고 Cowork 수기 유지** — 기각. 9/131 수기 실패가 반복되고, 전 종목 자동 수집(`backlog_fetch` 배치)의 취지와 모순된다.
- **유형1 구조파싱+검산 자동, 나머지 Cowork 폴백** (채택).

## Consequences

- **유형2(건설 공사진행 표: 수주총액·진행률·미청구공사, 기납품 없음)·유형4(다중엔티티 지주사, 한화)·시나리오 모호 표(삼바: 합계 없는 복수 데이터행)**는 검산/판단이 어려워 `source='pending'`으로 두고 Cowork가 채운다.
- **유형3(자동차·영업비밀·잔고 미기재)**는 행을 아예 생성하지 않는다(제외).
- 다중엔티티는 합계가 연결 총액이라 검산은 통과하지만 모회사 기준이 아니므로, 코드가 다중엔티티를 감지해 pending으로 보낸다(한화 연결 116조를 자동 저장하지 않음).
- 검산 컬럼이 없는 단일수치 표는 검산 불가 → pending(recall을 희생하고 정밀도 우선).
- `source` 3분류: `'dart'`(코드 자동, 검산 통과), `'llm'`(Cowork 수기), `'pending'`(미채움). ADR-0002의 document.xml 원문 수집 전략 자체는 유효하며, 본 ADR은 그중 "수치를 누가 채우는가"만 갱신한다.
- **금액은 억원으로 정규화해 저장**(`'dart'` 행 `unit='억원'`, 표 단위 ÷환산: 백만원÷100·천원÷10^5·조원×10^4). 프론트 `BacklogChart.fmtAmt`가 amount를 억원으로 가정(unit 미사용)하기 때문 — 정규화 없이는 단위 다른 종목 간 차트 비교가 깨진다. 표 단위 캡션을 못 읽으면(USD천·외화·줄바꿈 분리) 억원으로 폴백하지 말고 pending으로 떨군다(폴백 시 ×100 대형 오저장 발생).
