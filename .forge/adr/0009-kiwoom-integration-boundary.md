# 키움 REST API 연동 경계 — KR 전용·읽기전용·서버측 단일 자격증명

키움 REST API(208개 TR)를 PortfoliOn에 연동하면서 세 가지 경계를 못박는다. **(1) KR 전용**: 키움 API는 전부 "국내주식"(+금현물/ELW/ETF)이라 미국/해외 종목 데이터가 없다. US 시세·재무는 계속 yfinance로 받고, 키움은 KR 시장데이터 소스로만 쓴다. **(2) 읽기전용**: 시세·차트·수급·랭킹 등 조회 TR만 사용하고 계좌(kt00xxx)·주문(kt10000~)·신용주문·금현물주문 TR은 연동하지 않는다 — 실제 돈이 오가는 주문은 자격증명 유출·버그·오주문의 금전 리스크가 차원이 다르다. **(3) 서버측 단일 자격증명**: 키움 키(오너 개인계좌 65910753)는 유저별 OAuth 로그인이 아니라 백엔드의 단일 데이터 소스로 쓴다. `KIWOOM_APP_KEY`/`KIWOOM_SECRET_KEY`를 `.env.docker`(gitignore)에 두고, 토큰은 인프로세스 싱글톤으로 발급·갱신한다.

## Considered Options

- **키움으로 전체 시장데이터 일원화** — 불가. 키움엔 US 종목이 없어 yfinance를 대체할 수 없다. 포트폴리오가 US+KR 혼합이므로 듀얼 소스(US=yfinance, KR=키움)는 영구 구조다.
- **계좌·주문까지 풀 연동** — 거부. 읽기전용 시세 소스라는 목적에 비해 금전 리스크가 과도. 추후 별도 의사결정으로 다시 다룬다.
- **유저별 키움 OAuth** — 거부. 멀티유저 앱이지만 키움 데이터는 공용 시장데이터라 유저별 계정이 불필요. 단일 서버 자격증명이 단순하고 충분.

## Consequences

- KR 컨센서스/목표가/투자의견은 키움에 해당 TR이 **없으므로** FnGuide/Naver Research를 계속 쓴다. KR 상세 재무(매출·자산 등)도 Naver가 더 풍부해 유지(키움 ka10001은 PER/EPS/ROE/PBR만, 외부벤더·주1회 갱신).
- 키움 장애(토큰만료·레이트리밋·장마감 퀂)에 대비해 KR 시세는 **키움 우선 + Naver 폴백**으로 마이그레이션한다(첫 대체 한정, 검증 후 폴백 제거 여지).
- 실시간 스트리밍(WebSocket 0B/0D 등)은 별도 아키텍처(상시 연결·구독한도·프론트 푸시)라 후속 Phase에서 다룬다.
