# PortfoliOn — 도메인 용어집 (CONTEXT)

이 문서는 용어집입니다. 구현 세부사항이 아니라 "우리가 같은 단어로 같은 것을 가리키는가"만 다룹니다.

## 배치 (Batch)

데이터를 생성/갱신하는 **이름 붙은 작업**으로, 스케줄러 크론 잡 1개와 1:1 대응한다. 실행 1회(run)마다 **트리거 종류**를 가진다:
- **자동(auto)** — APScheduler 크론으로 백그라운드 실행 (`scheduler.py`)
- **수동(manual)** — 사용자가 설정화면 버튼으로 실행

한 배치는 두 트리거를 모두 가질 수 있다 (예: 리포트 생성·구루 크롤·컨센서스·레버리지·대차). 따라서 자동/수동은 배치의 고정 속성이 아니라 **실행(run)의 속성**이다. 하나의 크론이 여러 데이터셋을 만들어도 배치는 하나로 센다. 단, 한 크론이 만드는 데이터셋들이 **시장별로 깔끔히 갈라지면**([[배치 시장 분류]]) 시장별 형제 배치로 분리한다(daily_report·earnings·monthly).

현재 자동 배치: daily_report_kr·daily_report_us(2026-06 시장별 분리 — 아래 [[장마감]] 참조), earnings_kr·earnings_us(KR Top2 / M7 분리), monthly_kr·monthly_us(KR수출 / FRED 경제지표 분리), guru_crawl, daily_digest, leverage_fetch, lending_fetch, kr_rankings_fetch, us_rankings_fetch, investor_trend_fetch, short_sell_fetch, backlog_fetch.

## 배치 현황 (Batch Status)

모든 배치를 한 화면에서 보는 뷰. 배치별로 다음을 표시한다:
- **주기** — 실행 스케줄(크론)을 사람이 읽는 문구로
- **사용처** — 그 배치가 만든 데이터를 소비하는 UI 위치
- **다음 실행** — 다음 자동 실행 예정 시각 (APScheduler에서 실시간 조회)
- **실행 이력** — 최근 실행 기록(성공/실패), 각 줄에 자동/수동 태그

## 배치 시장 분류 (Batch Market Classification)

각 [[배치]]가 다루는 데이터가 어느 시장에 귀속되는지의 분류. **국내 / 해외 / 공통** 셋이며, [[배치 현황]] 허브의 시장 서브탭이 이 축으로 나뉜다(탭 안은 기존 category 섹션 유지). 판정 기준은 **데이터의 출처·주체 국가**다(사용처 시장이 아니다):

- **국내(KR)** — KR 소스/대상: DART·KOFIA·키움 기반 (backlog·leverage·lending·kr_rankings·investor_trend·short_sell·daily_report_kr·earnings_kr(KR Top2)·monthly_kr(KR수출)).
- **해외(US)** — US 소스/대상: yfinance S&P·M7·FRED (daily_report_us·us_rankings·earnings_us(M7)·monthly_us(FRED 경제지표)). FRED 경제지표는 양 시장 투자판단에 쓰는 매크로지만 **출처가 미 연준이라 해외**로 둔다(출처국 기준의 일관 적용).
- **공통(both/global)** — 한 단위(종목·유저) 안에서 KR·US가 엉켜 시장 귀속이 불가능하거나 글로벌인 배치: consensus(daily_report에 내장, 종목 market별 분기), daily_digest(유저 포트폴리오 KR+US 합산), guru_crawl(dataroma 글로벌).

데이터가 시장별로 깔끔히 갈라지는 배치만 시장별 형제로 **분리**한다(earnings·monthly는 한 크론이 시장별 독립 함수 2개를 호출하던 구조라 분리 가능). consensus·digest는 분리 불가라 공통에 남는다. 정본 경계·분리 규칙은 ADR-0013.
_Avoid_: "사용처가 양 시장이면 공통"(아니다 — 출처국 기준), "US 데이터는 다 해외"(맞지만 매크로/지수도 출처국으로 판정)

## 스케줄 패턴 (Schedule Pattern)

한 배치의 자동 실행 주기를 표현하는 형태. 4가지가 있다:
- **일배치(daily)** — 매일 지정 시각에 1회
- **주배치(weekly)** — 지정 요일(복수 가능)의 지정 시각
- **월배치(monthly)** — 매월 지정 일(day-of-month)의 지정 시각
- **인터벌(interval)** — 지정 시간대 안에서 N분마다 반복 (예: 장중 랭킹)

배치별로 패턴을 자유롭게 고를 수 있고, 편집은 admin [[액션]]만 가능하다(열람은 모두). **타임존은 패턴의 일부가 아니라 배치 고정 속성**이다(편집 불가, 기본 Asia/Seoul, us_rankings만 America/New_York). 인터벌엔 폭주 방지 가드레일이 있다(최소 5분, 중복 실행 방지).

## 장마감 (Market Close)

리포트 [[배치]]가 "오늘 데이터"를 잡는 기준 시점. 시장마다 다르다:
- **KR** — 정규장(KRX)은 15:30에 단일가로 공식 종가가 확정되지만, 이 서비스는 NXT(넥스트레이드) 연장세션(~20:00)까지 보는 통합(SOR) 시세를 쓰므로 **장마감 = 20:00**으로 본다. 일봉 종가는 15:30 KRX 종가로 동일하나, 스냅샷 **현재가**는 20:00 NXT 가격을 반영한다.
- **US** — 16:00 ET. 한국시간으로는 새벽 05:00(서머타임)~06:00(겨울)이라 KST 아침에 마감 직후가 된다.

**시장별 리포트 생성 시각**: 가장 신선한 데이터를 박제하려면 각 시장 장마감 직후에 그 시장 종목만 생성한다 — KR은 20:30 KST, US는 07:00 KST(둘 다 Asia/Seoul). 단일 08:00 배치는 US엔 신선했지만 KR은 개장 전이라 어제 종가가 박제되던 문제를 해소한다.

**기대 리포트 날짜**: 어떤 종목의 "최신 리포트가 있어야 할 날짜". 시장별·시각인지로 계산한다(KR은 20:30 전이면 직전 영업일, 후면 당일; US는 07:00 기준). 종목 최신 스냅샷이 이 날짜보다 오래되면 "미생성"으로 본다.

## 사용처 (Usage / Consumer)

한 배치가 만든 출력 데이터를 **소비하는 UI 위치**(페이지/탭). 예: `kr_rankings_fetch` → 랭킹 탭, `earnings_refresh` → Market Hub. 한 배치가 여러 사용처를 가질 수 있다.

## 실행 이력 (Execution Log)

배치 실행의 시작/종료/성공·실패/에러 기록. 배치별 **최근 N건(잡당 20건)**만 보관하며, 초과분은 insert 시 자동 삭제한다(별도 정리 크론 없음). 저장소는 `job_runs` 테이블.

## 액션 (Action) / 열람 (View)

- **열람** — 배치 현황을 보는 것. 권한 보유 사용자 누구나 가능(읽기 전용).
- **액션** — 스케줄 편집·지금 실행·백필 등 상태를 바꾸는 조작. **admin만** 가능(UI 숨김 + API 게이트).

## 미생성 (Ungenerated)

리포트 목록에서 **마지막 daily_report [[배치]] 스케줄일을 기준으로 뒤처진 종목**들의 탭/분류. 정본 정의는 "**최신 리포트 날짜가 그 스케줄일보다 이전이거나, 리포트가 아예 없는** 종목"이다. 즉 스케줄일보다 더 최신 리포트(예: 주말 수동 생성분)가 있으면 미생성이 **아니다**.
_Avoid_: "정확히 그 스케줄일의 리포트가 없으면 미생성"(틀림 — 더 최신 리포트가 있어도 미생성으로 오분류됨). ETF 여부와 무관(ETF도 일반 종목과 동일 기준).

## 그외 종목 (Other-User Stock)

관리자가 리포트 목록을 `scope=all`로 볼 때만 보이는 탭/분류. 정본 정의는 "**다른 사용자의 보유/관심(user_stocks)에는 있으나 관리자 본인의 목록엔 없는** 종목"이다(`is_mine=false`). 목록은 ticker 단위로 합산되므로 그외 한 항목이 여러 사용자에게 동시에 속할 수 있다. 관리자의 **삭제**는 그 ticker를 **모든 사용자의 user_stocks에서 제거**하는 것을 뜻한다(다른 사용자 항목 포함, 보유·관심 양쪽). 스냅샷(리포트 데이터)은 건드리지 않아 보이지 않는 고아로 남는다. 본인 목록엔 없으므로 그외 종목엔 수정·승격 액션이 의미가 없다(삭제만 노출).
_Avoid_: "내 목록에서만 제거"(틀림 — 본인 목록엔 애초에 없음), 리포트/스냅샷 삭제와 혼동(그외 삭제는 소유만 제거).

## 수주잔고 (Order Backlog)

정기보고서 "수주상황" 표의 **기말 잔여 수주액** — 아직 납품·매출로 인식되지 않은, 앞으로 실현될 수주분. 산식상 `수주총액 − 기납품액`(또는 `기초수주잔액 + 신규증감 − 기납품액 = 기말수주잔고`)이다. 다중엔티티 지주사(예: 한화에어로)는 공시 수주상황 표의 **연결 합계를 정본 수주잔고**로 하고, **[[사업부문 분해]]**로 모회사 본체분과 자회사분(한화오션·한화시스템 등)을 구분해 보여준다. (별도 상장 자회사가 섞여 생기는 이중계상은 종목 표시가 아니라 포트폴리오 합산 단계의 문제로 따로 다룬다.)
_Avoid_: 수주총액(누계라 다름), 백로그(영문 혼용)

## 수주총액 (Cumulative Order Total)

계약 체결 기준 **누계 총 수주금액**. 잔고가 아니라 누계이므로 수주잔고보다 항상 크거나 같다(예: 한화 162조 vs 수주잔고 116조). 자동 추출에서 이 값을 수주잔고로 잘못 잡으면 크게 과대계상된다.
_Avoid_: 수주잔고와 혼용

## 기납품액 (Delivered Amount)

수주총액 중 **이미 납품·매출로 인식된 금액**. `수주총액 − 기납품액 = 수주잔고`. 검산 게이트의 핵심 항.

## 사업부문 분해 (Segment Breakdown)

다중엔티티/다부문 종목의 [[수주잔고]]를 **사업부문(항공·방산·해양 등) 단위로 나눈 배열**. 각 부문은 1개 이상 법인의 합이며(예: 방산 = 한화에어로 방산 + 한화시스템 방산), 법인별 내역을 함께 보유한다. **부문 합 = 연결 수주잔고**(이 등식이 다중엔티티 자동추출의 검산 게이트). 모회사 본체와 자회사분을 한 화면에서 구분하기 위한 표현.

## 키움 시세 소스 (Kiwoom Market Source)

오너의 키움 개인계좌 자격증명(서버측 단일 키)으로 받는 **KR 시장데이터** 소스. "키움 연동"은 이 한정된 의미다 — 유저별 키움 로그인도, 계좌·주문 연동도 아니다. 경계는 셋: **KR 전용**(키움엔 US 종목이 없어 US는 계속 yfinance), **읽기전용**(시세·차트·수급·랭킹 조회만, 계좌·주문 TR 미사용), **공용 데이터**(유저 무관, 시장 공통). KR 컨센서스/목표가/상세재무는 키움에 없거나 빈약해 기존 소스(FnGuide·Naver)를 유지한다. 정본 경계는 ADR-0009.

**KR 시세의 기본 venue는 통합(SOR `_AL`)** — 현재가·차트를 통합 코드로 받아 KRX+넥스트레이드(NXT)의 합성 BBO 통합호가가 되고, NXT 확장시간(08:00~20:00 KST)까지 포함한다(비-NXT 종목은 통합이 KRX로 자동 폴백). 즉 "KR 시세"는 KRX 단독이 아니라 통합값을 가리킨다(랭킹은 Naver 유지·미채택, 수급은 KRX 기반). 정본은 ADR-0010.
_Avoid_: "키움 연동 = 모든 시세를 키움으로"(US는 불가), "내 계좌 연동"(읽기전용 시세만, 잔고/주문 아님)

## 백업 시세 소스 (KIS / 한국투자증권)

한국투자증권(KIS) REST API로 받는 **읽기전용 백업 시장데이터** 소스. [[키움 시세 소스]]와 같은 "서버측 단일 자격증명·공용 데이터·읽기전용" 성격이지만 둘이 다르다: **(1) 1차가 아니라 백업** — 현재가 조회의 1차 소스(KR=키움, US=yfinance)가 실패할 때만 발동하는 폴백이다(체인 KR=키움→KIS→Naver, US=yfinance→KIS). **(2) KR+US 둘 다 커버** — 키움이 KR 전용인 것과 달리 KIS는 국내·해외주식을 모두 제공해 US 폴백도 된다. 단 US 시세는 기본 15분 지연·주요지수 구성종목 중심이라 "백업"으로만 충분하다(실시간·전체커버는 유료/별도). 주문·계좌 TR은 키움과 동일하게 **연동하지 않는다**(읽기전용). **자격증명 미설정이 안전 기본값** — 서버에 키가 없으면 KIS는 휴면하고 기존 체인(키움/Naver/yfinance)만 동작하며, 키를 주입하면 폴백이 활성화된다. 정본 경계는 ADR-0011.
_Avoid_: "KIS = 1차 시세"(백업이다), "KIS로 계좌·주문 연동"(읽기전용 시세만), "US도 실시간"(기본 15분 지연)

## 공매도 추이 (Short-Sell Trend)

종목별 **일별 공매도 *거래*** 시계열 — 그 날 체결된 공매도 거래량·거래대금과 전체 거래 대비 **공매도 비중(%)**. 키움 ka10014([[키움 시세 소스]]의 읽기전용·KR 전용 경계 안)로 받는다. **공매도 *잔고*(미상환 누적 공매도, KRX가 T+2로 별도 공시)와 다르다** — ka10014는 *거래* 추이지 잔고가 아니므로, 잔고/잔고비중은 응답에 없으면 키움 범위 밖(KRX 등 별도 소스 후속). 리포트 상세 "기술·수급" 하위탭에서 [[수급 추이]] 옆에 per-stock으로 표시.
_Avoid_: 공매도 잔고(미상환 누적분 — 별도 데이터셋이라 ka10014에 없을 수 있음), 대차잔고(차입 잔고 = 금융위 lending 지표로 공매도의 원천이나 별개)

## 공시 피드 (Disclosure Feed)

보유·관심 KR 종목의 **DART 원시 공시 목록**을 자동 수집한 시계열 — 공시 일자·제목(report_nm)·유형(pblntf_ty)·접수번호(rcept_no, DART 뷰어 링크). DART `list.json`(corp_code별)로 받으며 [[수주잔고]]와 같은 DART 연동(corp_code 매핑·`DART_API_KEY`)을 재사용한다. **핵심 유형만** 수집한다 — A(정기·실적)·B(주요사항: 유증·자사주·합병·대규모계약·CB)·C(발행)·D(지분: 대량보유); E·F·I·J(기타·외감·거래소·공정위) 노이즈는 제외. KR 전용·읽기전용. 리포트 상세 "최신 공시" 섹션 + 일일 다이제스트(=일일 텔레그램 전달)에 노출.
_Avoid_: **`recent_disclosures`와 혼동 금지** — 그건 외부 Cowork가 쓰는 **애널리스트 코멘터리(분석 텍스트)**이고, 공시 피드는 **자동 수집한 원시 공시 목록**이다. 둘은 별도 저장소(피드는 `stock_disclosures` 테이블, 코멘터리는 `tickers.recent_disclosures`)이며 공시 피드는 코멘터리를 덮지 않는다.

## 뉴스 (News)

보유·관심/랭킹 종목의 **언론 기사 헤드라인**(제목·링크·출처·시각) — `scraper.get_news`가 시장별 라이브 스크레이프(KR=Naver, US=yfinance). 링크 기준 중복제거·최신순. **전용 저장 테이블 없음** — 리포트 상세는 요청 시 **라이브 fetch**(스냅샷 `data.news`는 폴백), 다이제스트는 아침 리포트가 박제한 스냅샷 news를 **읽어** 노출(배치 스크레이프 없음).
_Avoid_: **[[공시 피드]](DART 원시 공시)·`recent_disclosures`(Cowork 애널리스트 코멘터리)와 혼동 금지** — 셋 다 리포트 상세 "최근 공시 & 뉴스" 근처에 병치되지만 출처·저장소가 전부 다르다: 뉴스=언론 기사(스크레이프·무저장), 공시 피드=DART 규제 공시(`stock_disclosures`), recent_disclosures=AI 요약 텍스트(`tickers`).

## 매크로 신호 (Macro Signals)

경기·신용·유동성을 가늠하는 **FRED 거시 시계열 + 그 해석 신호** — 장단기 금리차(T10Y2Y; <0=수익률곡선 역전=침체 경고), 하이일드 스프레드(BAMLH0A0HYM2; 급확대=신용 스트레스), M2(유동성), 기준금리(DFF). 기존 [[경제지표]](CPI·실업률)와 같은 FRED `_fetch_series` 패턴·`market_cache` 저장을 재사용하되, 금리차·HY는 **일간 갱신**이라 별도 일배치(`macro_signals_fetch`)다. 출처국 미국이라 시장 분류는 **해외**(FRED=US, [[배치 시장 분류]] 일관). MarketHub 시장지표 탭에 노출.
_Avoid_: **매크로 *상관*(Macro Correlation)과 혼동 금지** — 그건 포트폴리오 분석탭 MacroTab의 **보유 종목–매크로 자산(TLT/UUP/USO/VIX) 90일 상관관계**(`analysis_service.MACRO_TICKERS`)이고, 매크로 *신호*는 **FRED 거시지표의 침체/신용 신호 시계열**이다. "매크로" 한 단어에 두 다른 기능이 걸려 있다.

## 시장지수 밸류에이션 (Market-Index Valuation)

시장 *전체*가 역사적으로 비싼지 가늠하는 **지수 단위 멀티플** — 현재는 US S&P500 Shiller CAPE(경기조정 PER: 최근 10년 실질이익 평균 대비)의 현재값+역사 통계(평균/중앙/최소/최대)뿐이다. 함께 묶이는 **시장지수 레벨**(KOSPI·KOSDAQ·S&P500 일간 레벨·등락)은 포트폴리오의 벤치마크 데이터다. 둘 다 [[매크로 신호]]·fx처럼 MarketHub 시장지표 탭에 노출.
_Avoid_: **per-종목 밸류에이션(PER/PBR/목표가)과 혼동 금지** — 그건 *개별종목*이 싼지 비싼지고, 시장지수 밸류에이션은 *시장 전체*의 과열도다. [[매크로 신호]](FRED 거시 침체/신용 신호)와도 다른 축. 출처 주의: **FRED엔 주식 CAPE 시리즈가 없다**(FRED "Case-Shiller"는 *주택가격* 지수) — `multpl.com` 크롤이 무료 출처다.

## 시장 센티먼트 지수 (Fear & Greed)

시장 전체의 투자심리를 **0~100 단일 수치**로 나타내는 CNN 공포·탐욕 지수(0=극단적 공포, 100=극단적 탐욕) + rating(공포/중립/탐욕). **US 시장 대상**(KR 무료 F&G 소스 부재 — [[시장지수 밸류에이션]]의 S&P CAPE가 US 전용인 것과 같은 결). CNN **비공식** 엔드포인트라 전체 브라우저 헤더(UA+Accept+Origin+Referer+sec-ch-ua)가 있어야 응답하고(불완전하면 418), 언제든 막힐 수 있어 **실패 시 직전 저장값으로 graceful**. [[매크로 신호]]·VIX처럼 시장지표 탭에 노출.
_Avoid_: **VIX(변동성지수)와 혼동 금지** — VIX는 옵션 내재변동성 단일 지표(프론트가 관례적으로 "공포지수"로 라벨)이고, Fear&Greed는 7개 하위지표(모멘텀·주가강도·주가폭·풋콜·변동성·안전자산·정크본드 수요) **합성** 0~100 심리지수다. [[매크로 신호]](FRED 거시 침체/신용 신호)와도 다른 축(그건 거시지표, 이건 시장 투자심리).

## 다음날 코스피 신호 (Next-day KOSPI Signal)

전 거래일 KR 장마감 후의 **오버나잇 해외 시장 움직임**(S&P500·나스닥 등락 + USD/KRW 변화)을 합성해 **다음 거래일 코스피의 방향**을 강세/중립/약세로 예측하는 지표. 매 거래일 개장 전(08:30 KST) 신호를 시점 박제하고, 그날 코스피 실제 결과(시가 갭·종가 등락)와 대조해 **적중률을 누적**한다(라이브 체크). 코스피 *선물*이 반영하는 것이 결국 미국 오버나잇+환율이라, 이 프록시로 방향을 예측한다(실제 선물 데이터는 [[코스피200 선물]]로 별도 표시). **실제 (야간)선물 오버나잇을 이 신호의 드라이버로 편입하는 안은 검토 후 기각** — KRX 야간선물(18:00~06:00)이 KIS 시세 TR에 노출되지 않아(2026-07-08 야간세션 중 라이브 프로브로 확인: `FHMIF10000000`이 정규장 종가만 반환) 08:30 배치 시점 선물가는 어제 종가라 오버나잇 신호가 0이고, 그 오버나잇도 대부분 이미 쓰는 미국 프록시와 겹친다. 소스는 US지수+환율이나 대상이 코스피라 [[배치 시장 분류]]는 국내(KR). MarketHub 시장지표 탭에 노출.
_Avoid_: **[[매크로 신호]](FRED 거시 절대신호)·매크로 상관(보유종목–매크로 자산 상관)과 혼동 금지** — 셋 다 "신호/매크로"지만 이건 *다음날 코스피 방향* 예측이다. 실제 [[코스피200 선물]](KIS 선물 일봉 차트)과도 구분 — 이건 선물이 아니라 미국 오버나잇 프록시다.

## 코스피200 선물 (KOSPI200 Futures)

KRX 상장 KOSPI200 지수선물의 **최근월물 일봉 시계열 + 현재 베이시스**를 시장지표 탭에 표시하는 정보성 지표. [[백업 시세 소스]](KIS)의 국내선물 시세 TR로 받으며(경계 확장 ADR-0022), 최근월물 코드는 공식(`"A01"+연%10+월`, 분기월)으로 계산하고 만기는 응답 `futs_last_tr_date`로 롤오버한다. **베이시스**(선물−현물 = 콘탱고/백워데이션)가 선물-고유 인사이트다. KR 유일 소스가 KIS라 키 미설정 시 휴면.
_Avoid_: **[[다음날 코스피 신호]]와 혼동 금지** — 그건 선물이 아니라 *미국 오버나잇 프록시*로 다음날 방향을 예측하는 것이고, 이건 *실제 KOSPI200 선물*의 일봉/베이시스 표시다(정보성, 예측 아님). 최근월물은 **연속선물이 아니다** — 분기마다 코드가 바뀌므로 증분 병합/스티칭이 계약을 뒤섞는다(요청마다 전체 윈도우 fetch).

## 데이터 소스 (Data Source)

한 [[배치]]가 데이터를 **읽어오는 외부 출처(fetch origin)** — 키움·KIS·yfinance·Naver·FnGuide·FRED·KOFIA·DART·관세청·dataroma 등. `batch_registry`의 `source` 필드(list)로 갖고 [[배치 현황]] 카드에 표시한다. **[[사용처]](Usage)와 반대 방향**이다 — 사용처는 그 배치가 *만든* 데이터를 **소비하는 UI 위치**, 데이터 소스는 그 배치가 *읽어오는* **외부 API**다. 정적 메타데이터라 배치의 fetch 소스가 바뀌면 `source`도 함께 갱신해야 한다(CLAUDE.md 규칙·DoD).
_Avoid_: [[사용처]](소비 UI)와 혼동 — 방향이 정반대다(소스=입력 출처, 사용처=출력 소비처).

## 단계 라벨 (Stage Label)

리포트 "장기 성장 계획"의 각 **이니셔티브가 어느 진행 단계에 있는지 보여주는 짧은 문구**(칩으로 표시). 이 문구는 **종목 도메인에 맞게 작성자(외부 Cowork)가 기입하는 자유문구**다 — 제약은 "임상3상", 반도체는 "양산 램프업", SaaS는 "정식 출시"처럼 산업마다 다르다. **`status`(launched/phase2/phase3/announced/completed)와 구분**한다: `status`는 칩 **색상**을 정하는 거친 진행도 신호일 뿐 표시 단어가 아니다. 산업별 마일스톤 어휘는 무한하므로 고정 enum이나 섹터 추론으로는 정확히 못 담고, 맥락을 아는 작성자가 단어를 정해야 정확하다(고정 enum/섹터 분기는 반례를 못 막음 — 제약사의 *비*임상 이니셔티브를 "임상"으로 오표기 등).
_Avoid_: **`status`를 단계 라벨로 혼동 금지** — 과거엔 프론트가 `phase3`→"임상3상"으로 하드코딩해 모든 종목(예: 삼성전자)에 임상 라벨이 잘못 붙었다. 이제 라벨은 작성자 기입 문구가 정본이고, status는 색상용 폴백이다.

## 수급 스코어 (Supply/Demand Score)

보유·관심 KR 종목별로 **종목 단위 수급 데이터를 합성해 매긴 우호/중립/경계 3단계 밴드** + 그 근거 플래그 묶음. 입력은 **per-종목 데이터 둘뿐** — [[공매도 추이]](`market_short_sell`: 공매도 비중·잔량)와 외국인/기관 수급(`market_investor_trend`: 외인+기관 순매수·외인보유율)이다. **신용잔고·대차잔고는 시장 전체 집계라 종목 점수에서 제외**하고, Market 허브의 시장 백드롭으로만 둔다(입자 불일치 — 시장 단위 값을 개별 종목 점수에 투영하면 변별력이 사라지고 오도된다). 점수 형태는 **단일 수치가 아니라 플래그→밴드**다: '공매도 비중 급증'·'외인·기관 동반 순매도' 같은 근거 보이는 플래그를 최근 5거래일 vs 직전 20거래일 *상대* 비교로 켜고(가중치 없음), 경계 플래그 개수로 밴드를 판정한다(균형 규칙: 경계 플래그 ≥2 또는 강한 단일 → 경계). `supply_score_fetch` 일배치가 사전계산해 `stock_supply_score`에 저장하고, 대시보드 배지·리포트 상세 기술·수급 탭은 **저장값만 읽는다**(요청경로 계산 0). KR 전용 — US 종목·결측은 '해당 없음'. 정본 경계는 ADR-0014.
_Avoid_: **[[매크로 신호]]·시장 수급지표(신용/대차)와 혼동 금지** — 그건 시장 전체 집계 시계열이고, 수급 스코어는 *종목별* per-stock 신호다. 단일 0~100 점수 아님(플래그+밴드). [[데이터 소스]] 표기상 출처는 키움(per-stock 시세 파생)이다.

## 지분공시 신호 (Insider / Major-Holder Trading Signal)

보유·관심 KR 종목의 **내부자·대량보유자 지분 변동**을 DART 상세 공시에서 구조화해 뽑은 신호 — 누가(보고자·직위), 몇 주를(증감주식수 +/−), 거래 후 보유비율(%)을, 언제. 두 출처: **임원·주요주주 소유상황보고**(`elestock.json` — 내부자 1주 단위 매매)와 **주식등의 대량보유상황보고(5%룰)**(`majorstock.json` — 큰손 진입/이탈·행동주의). DART 연동(corp_code 매핑·`DART_API_KEY`)과 [[공시 피드]]의 수집·배치 패턴을 재사용한다. **per-종목 순매수/순매도 신호**는 최근 N일 증감주식수 합으로 파생한다(저장 line item에서 SQL 집계, 요청경로 라이브 호출 0). KR 전용·읽기전용. 리포트 상세 '내부자·5% 지분변동' 섹션 + 종목 신호 배지 + 일일 다이제스트에 노출.
_Avoid_: **[[공시 피드]]와 혼동 금지** — 공시 피드는 DART `list.json`의 **원시 공시 제목 목록**(D 지분 유형을 포함하나 제목·접수번호만)이고, 지분공시 신호는 `elestock`/`majorstock` **상세를 파싱한 구조화 거래 line item + 순매수 파생**이다(한 층 더 깊다). **[[수급 스코어]]와도 다름** — 수급 스코어는 외국인/기관·공매도 *거래흐름* 합성 밴드, 지분공시 신호는 *내부자·대량보유자가 보고한 지분 변동*이다. 신호 배지 색은 KR 가격 토큰(`success`/`danger`) 금지·전용색([[수급 스코어]] SupplyBadge 규약과 동일).

## 추천 (Recommendation)

보유·관심 여부와 무관하게 **"지금 살펴볼 종목"을 합성 점수로 골라 제시**하는 기능. 세 산출이 한 화면(Research 허브 '추천' 탭)에 모인다:
- **발굴(Discovery)** — 내가 아직 보유·관심에 안 담은 종목 중 점수 상위. [[랭킹]]·관심종목과 다르다: 랭킹은 시장 전체 *거래* 상위(중립적 사실)이고, 발굴은 *내가 안 가진* 종목을 *합성 점수*로 골라준 것이다.
- **관심 재정렬** — 내 관심종목을 추천 점수순으로 정렬(무엇을 먼저 볼지).
- **보유 액션** — 내 보유종목에 추매/익절/홀딩 신호(추천 점수 + 내 포지션 맥락).

점수는 **멀티팩터 합성 0~100**(밸류=목표가 상승여력·모멘텀·스마트머니)이고, "왜 추천"은 [[수급 스코어]]·[[지분공시 신호]]처럼 **정량 플래그**로 보인다(백엔드 LLM 0). 세 섹션은 저장된 *글로벌* 점수를 요청 시 그 유저의 보유/관심으로 partition한 것이다(점수는 종목 공유, 섹션은 per-user). 정본 경계는 ADR-0015·0016(결측군 합성은 0016이 중립 채움으로 대체).
_Avoid_: "[[랭킹]] = 추천"(랭킹은 거래 상위 *사실*, 추천은 점수 *선별*), "발굴이 AI가 쓴 분석"(근거는 정량 플래그; AI 내러티브는 [[딥다이브]]로 별도·비동기), "[[수급 스코어]] = 추천"(수급은 *한 축*의 종목 밴드, 추천은 *여러 축 합성* 선별 — 수급 스코어가 추천의 한 입력일 수 있다).

## 발굴 유니버스 (Discovery Universe)

[[추천]] 점수를 매기는 **후보 종목 집합**. 최종 목표는 전체 시장(~2700 KR + 503 US)이나 **점진 확장**한다 — v1은 KR 시총 상위 ~200 + US S&P500 + 전 유저 추적종목(+US 구루 보유). 비용 통제를 위해 **2단 깔때기**로 좁힌다: 싼 신호로 후보를 거른 뒤, 후보에만 비싼 *리치* 신호(OHLC 모멘텀·컨센서스·KR 수급/지분공시)를 붙인다. 후보 선별은 **KR만 시총 랭킹**(시총 상위 N — 성장 다이얼)이고 **US는 S&P500 전량 + 구루 보유**(S&P500 자체가 시총 큐레이션 — US 개별 시총 데이터는 없다), **추적종목은 무조건 후보 통과**다(ADR-0021). 구루 신호는 **US 전용**(dataroma 13F). 항상 전 유저 추적종목을 포함해 [[추천]]의 관심 재정렬·보유 액션이 점수를 잃지 않게 한다.
_Avoid_: "리치 신호를 전 종목에"(비용상 불가 — 깔때기 후보만 enrich), "시총 랭킹이 US에도 적용"(US 행엔 시총이 없어 랭킹이 무의미했고 그게 커버리지 붕괴의 근본원인 — ADR-0021), "구루를 KR에"(US 전용), "유니버스 = 추적종목"(추적종목은 부분집합 — 발굴은 미추적까지), "구루 유니버스 = 우량주만"(US 13F엔 저유동성 OTC/외국 티커가 섞여 있고, 거래량 급증 모멘텀과 결합하면 발굴 상위로 새므로 — CFRHF·HKHHF 등 라이브 UAT 확인 — 저유동성 종목은 점수·저장은 유지하되 발굴(discovery)에서만 제외된다(관심/보유 섹션엔 직접 담으면 그대로 노출), task#68).

## 딥다이브 (Deep-dive)

[[추천]] 종목의 **AI 분석을 요청**하는 동작. 백엔드엔 LLM이 없고 분석 텍스트는 외부 Cowork가 enrich API로 쓰므로, 딥다이브 = **그 종목을 관심종목에 추가**하는 위임이다 → 기존 "추적 시 Cowork 분석" 흐름이 다음 Cowork run에 *비동기로* 채운다(즉시 생성 아님). 후보마다 AI를 돌리지 않아 Cowork(외부 LLM) 비용이 사용자 행동에만 든다.
_Avoid_: "딥다이브 = 즉시 AI 리포트 생성"(비동기 — 백엔드가 Cowork를 못 돌림), "추천이 분석 큐를 신설"(관심추가로 위임 — 큐 없음).

## 사용자 목표가·손절가 (User Target / Stop)

사용자가 보유 종목에 **직접 설정**하는 익절 목표가와 손절 기준가. **애널리스트 컨센서스 목표가**([[추천]]·리포트가 쓰는 `daily_consensus_mart`/스냅샷 `target_mean` 등, 외부 증권사 집계)와는 **다른 축**이다 — 컨센서스는 시장의 예측이고, 사용자 목표가/손절가는 그 유저의 매매 계획이다. `user_stocks`에 per-유저로 저장하며(종목 공유 마스터 `tickers`·스냅샷 아님), 종목 native 통화(KR=원·US=달러, `avg_cost`와 동일 축). 대시보드 카드에 현재가 대비 거리(%)로 표시된다.
_Avoid_: "목표가 = 컨센서스 목표가"(별개 — 사용자 설정 vs 애널리스트 집계), "손절가를 컨센서스/시스템이 채운다"(사용자만 설정), "target_price를 tickers·스냅샷에 저장"(user_stocks per-유저).

## 노출·집중도 (Exposure / Concentration)

보유 포트폴리오가 **특정 섹터·통화(시장)·단일종목에 얼마나 쏠려 있는지**를 비중으로 본 것. 세 축이다: **통화 노출**(KR/US 시장별 비중), **섹터 노출**(섹터별 비중), **단일종목 집중도**(종목별 비중 내림차순 + top-N 집중비율 + 임계 초과 경고). 여기에 **포트폴리오 베타**(베타가중 노출 = Σ(비중×베타) = 포트 시장민감도)를 정보성 지표로 더한다 — 베타를 아는 보유만의 가중평균(가중치 재정규화)이라 **커버리지%**를 함께 표기하고, per-종목 beta 결측을 대비해 배치가 사전계산해 둔 저장값(`stock_beta`)만 읽는다. 비중 기준은 [[사용자 목표가·손절가]]가 아니라 리밸런싱과 같은 **전체-포트 KRW 환산 분모**다. **보유(holding)만** 대상 — 관심종목은 아직 담지 않은 자산이라 제외한다. 섹터를 모르는 종목(리포트 미생성 US·업종 역인덱스 배치 전 KR)은 `기타`로, FX 환산 불가한 US 종목은 집계에서 제외하고 별도 표기한다. Portfolio 분석 하위탭 '노출'에 표시.
_Avoid_: **종목별 리스크 서술(`tickers.risks`, Cowork 내러티브)과 혼동 금지** — 그건 한 종목의 정성 리스크 텍스트고, 노출·집중도는 *포트폴리오 전체*의 쏠림을 정량 비중으로 본 것이다. [[수급 스코어]](종목 수급 밴드)·[[매크로 신호]](시장 거시)와도 다른 축이다. 경고 배지 색은 KR 가격 토큰(`success`=빨강/`danger`=파랑) 금지·전용색([[수급 스코어]] SupplyBadge 규약과 동일).

## 거래소 코드 (Exchange Code)

KR 종목이 어느 거래소 소속인지를 가리키는 **앱 내부 정본 코드**: `KS`=KOSPI, `KQ`=KOSDAQ, `''`(빈 문자열)=비-KR(US 등). 종목 저장(user_stocks·tickers)·[[랭킹]]·[[추천]] 응답·MarketBadge 표시·yfinance 폴백 심볼(`.KS`/`.KQ`)이 **이 코드 하나로** 시장 세분을 표현한다. Naver marketValue API의 `stockExchangeType.code`가 이미 `KS`/`KQ`를 그대로 주므로 별도 변환이 없고, Naver 검색 API는 `typeCode`(KOSPI/KOSDAQ/ETF)를 `KS`/`KQ`로 매핑한다([[키움 시세 소스]]가 아니라 Naver가 출처). [[딥다이브]](관심추가)·[[랭킹]]에서 종목을 watchlist에 담을 때 이 코드가 정확해야 KOSDAQ이 KOSPI로 오표시되거나 yfinance 폴백이 `.KS`로 틀리지 않는다.
_Avoid_: "KS = 한국 전체"(KS=KOSPI 한정 — KOSDAQ은 KQ), "exchange는 표시용일 뿐"(시세 주경로는 6자리라 무영향이나 yfinance 폴백 심볼엔 영향 — `wrong < missing`), "비-KR도 코드를 채운다"(US는 빈 문자열).

## 배당 스케줄 (Dividend Schedule)

보유·관심 종목의 **다가오는 배당 일정 타임라인**: 종목별 배당락일·(US)지급일·주당배당액·(보유)예상 수령액을 날짜순으로 본 것. 리서치 허브 '배당' 탭에 목록으로 표시한다. 데이터는 [[배치]](`dividend_fetch`)가 종목별 yfinance `t.dividends` 이력에서 사전계산해 `stock_dividend_schedule`에 저장하고, 뷰는 저장값만 read한다(요청경로 라이브 fetch 금지).
_Avoid_: **대시보드의 연 예상배당(`total_expected_annual_income_krw`, [[배당]] 연 DPS 기반)과 혼동 금지** — 그건 *1년 합계 금액*이고, 배당 스케줄은 *언제 얼마가 들어오는지의 시점 타임라인*이다. 캘린더의 배당락 이벤트와도 다르다(캘린더는 exact-only·단일 이벤트, [[배당 예상]] 참조).

## 배당 예상 (Dividend Projection)

[[배당 스케줄]]에서 미래 배당락일을 **이력의 배당 주기(분기/반기/연)로 투영한 추정 이벤트**. yfinance는 미래 배당일을 신뢰성 있게 주지 않으므로(특히 KR — `t.calendar` 값이 과거인 경우 다수), 이력 기반 투영이 forward 커버리지의 유일한 수단이다. '**예상**' 라벨로 명시 표시하며 예상금액=직전 실금액. US는 `t.calendar`로 최근접 건을 '**확정**'으로 보강한다. 단 **US 확정은 종목별 `t.calendar` 신선도에 의존** — 미래 ex-date가 신선한 종목만 확정(예: MSFT), stale(과거값)이면 KR과 동일하게 예상 폴백(예: AAPL·KO). 즉 '확정' 부재는 버그가 아니라 그 종목 calendar가 아직 미래 배당일을 안 준 것.
_Avoid_: **캘린더의 배당락일과 혼동 금지** — 캘린더(`routers/calendar.py`)는 task#119 이후 **exact-only**(이력 추정 금지)를 유지한다. Projection은 오직 [[배당 스케줄]] 뷰에만 존재한다. 목적이 다르다: 캘린더=정확성, 배당 목록=forward 커버리지(ADR-0023).

## 배당 (Dividend)

종목의 배당 데이터. 두 층으로 나뉜다: **연 층**(`stock_dividends`, 연 주당배당·시가배당률 — US=yfinance info, KR=DART alotMatter; 대시보드 `yield_on_cost`·연 예상수령액 계산용) vs **스케줄 층**([[배당 스케줄]], 시점별 배당락/금액). 둘은 소스·저장·소비처가 다르다.

## Peer 할인/할증 (Peer Discount/Premium)

내 종목의 밸류에이션 멀티플(PER/PBR/PSR/EV/EBITDA)이 **경쟁사(peer) 중앙값 대비 몇 % 낮은지(할인)/높은지(할증)** — "싼지 비싼지"의 동종업계 비교 기준점. peer 집합은 리포트의 경쟁사 목록에서 자기 자신(is_self)을 제외한 것이고, 해당 지표 값이 있는 peer가 **2개 미만이면 판정하지 않는다**(wrong<missing). 기준은 평균이 아니라 **중앙값** — peer가 2~5개 소표본이라 outlier 1개에 평균이 쉽게 오염된다. 리포트 상세 경쟁사 섹션 상단에 지표별 칩으로 노출.
_Avoid_: **[[시장지수 밸류에이션]]과 혼동 금지** — 그건 시장 *전체*의 역사적 과열도(지수 단위 CAPE), 이건 *개별종목 vs 동종업계*의 상대 위치다. 또한 할인=**싸다는 신호일 뿐 "좋다" 판정이 아니다**(저평가엔 이유가 있을 수 있음 — 판단 보조 정보로만).

## 핵심 자원 (Key Resource)

업종마다 다른, 그 회사 경쟁력의 **원천이 되는 핵심 투입자원 + 그 자원의 정량 건강도 + 유지 동력**을 외부 Cowork가 분석해 리포트에 싣는 enrich 필드(`tickers.key_resource`, moat·[[단계 라벨]]과 같은 Cowork 작성 계열). IT·SW·컨설팅은 **인력**이 대표 예 — 직원수·1인당 매출·1인당 영업이익의 **분기별 추이(series)**와 그 인력을 붙잡는 **유지 동력(리텐션 인센티브: 스톡보상·이직률 등)**을 본다. 업종이 바뀌면 자원도 바뀐다(반도체·하드웨어=생산능력/가동률·CapEx, 제약·바이오=R&D 파이프라인, 자동차·EV=생산능력, 에너지·인프라·건설=[[수주잔고]]). 업종별로 "무엇을 핵심자원으로 보고 어떤 지표를 채울지"는 Cowork 가이드(`CLAUDE_COWORK_API.md`)의 업종별 표가 지침하며, 그 표는 [[단계 라벨]] 어휘 표와 **같은 5개 업종 버킷에 정렬**한다. 정량 수치는 **AI가 조사·기입**한다(백엔드 자동계산 아님 — 분기별 직원수의 안정적 소스가 없어 재무÷직원수 자동계산이 불가). 리포트 심층분석 탭에서 경제적 해자(moat) 옆에 표시.
_Avoid_: **경제적 해자(moat)와 혼동 금지** — moat는 경쟁우위가 *왜 지속되는가*(특허·브랜드·규모의 진입장벽)이고, 핵심 자원은 그 업종의 *핵심 투입자원이 무엇이며 정량적으로 건강한가*(인력이면 1인당 생산성·이직률)다. 인접하지만 축이 다르다. [[수주잔고]]와도 다름 — 에너지/건설에선 수주잔고가 핵심자원 지표의 하나로 *인용*될 수 있으나, 핵심자원은 그 값의 DART 자동추출이 아니라 Cowork의 업종 해석 서술이다.

## 주식 용어집 (Stock Glossary)

**최종 사용자에게** 어려운 주식 용어(PER·매물대·베이시스 등)를 설명하는 앱 내 용어 사전. 정본은 프론트 정적 모듈(용어→설명 맵, DB/API 없음)이고, 노출은 두 방식이다: ① 리포트 상세의 AI 자유 텍스트에서 **자동 매칭**된 단어(밑줄+클릭), ② 차트 **범례/지표 라벨**에 명시 배선된 용어. 클릭하면 단어 근처 **앵커 팝오버**(레이어)로 설명이 뜬다. 별도 브라우즈 화면은 없다(팝업 전용).
_Avoid_: **개발 용어집(.forge/CONTEXT.md — 이 문서)과 혼동 금지** — 그건 팀의 도메인 언어 정본이고, 주식 용어집은 최종 사용자 대상 학습 콘텐츠다. "용어집 = 별도 페이지"(아니다 — 인라인 팝오버 전용).

# Coding principles

You are a lazy senior developer. Lazy means efficient, not careless. The best code is the code never written.

Before writing any code, stop at the first rung that holds:

1. Does this need to be built at all? (YAGNI)
2. Does the standard library already do this? Use it.
3. Does a native platform feature cover it? Use it.
4. Does an already-installed dependency solve it? Use it.
5. Can this be one line? Make it one line.
6. Only then: write the minimum code that works.

Rules:

- No abstractions that weren't explicitly requested.
- No new dependency if it can be avoided.
- No boilerplate nobody asked for.
- Deletion over addition. Boring over clever. Fewest files possible.
- Question complex requests: "Do you actually need X, or does Y cover it?"
- Pick the edge-case-correct option when two stdlib approaches are the same size.
- Mark intentional simplifications with a `ponytail:` comment naming the ceiling and upgrade path.

Not lazy about: input validation at trust boundaries, error handling that prevents data loss, security, accessibility, anything explicitly requested.
